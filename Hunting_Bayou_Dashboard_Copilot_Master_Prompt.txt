PROJECT TITLE
Bayou Feedback: A Dutch-Informed, Citizen-Facing Waterway Response Dashboard for Hunting Bayou, Houston

PURPOSE
Build a public-facing, read-only dashboard that makes technical flood-control behavior more understandable for Houston residents. The focal corridor is Hunting Bayou in northeast/central Houston. This is not an emergency warning product and it must not claim that an infrastructure intervention causes a specific reduction in flood risk without an appropriate hydraulic or before/after analysis.

The dashboard should translate a principle observed during a University of Houston Netherlands resilience trip: Dutch water management treats flood resilience as coordinated, monitored intervention that preserves normal function, temporarily stores or redirects water where needed, and responds when conditions change. Relevant trip contexts were the Delta Works barriers at Deltapark Neeltje Jans and Maeslantkering, Kinderdijk, Delft University of Technology, Delfland Water Authority, and Living Lab Scheveningen.

The Houston translation is:
- Focus on an under-discussed but technically important waterway corridor rather than a famous city-center bayou.
- Display live and historical observations in language ordinary residents can understand.
- Explain how stormwater detention and conveyance improvements are intended to work.
- Use open government data to show observed waterway response, not to issue emergency instructions or certify infrastructure performance.

FOCAL HOUSTON CORRIDOR
Use Hunting Bayou as the default and only fully implemented corridor in v1.

Verified contextual facts to represent in a short “About this waterway” panel:
- Hunting Bayou is in the Houston/Galena Park/Jacinto City area and drains southeast to the Houston Ship Channel.
- The watershed is highly urbanized and includes Herman Brown Park as a major public/open-area context.
- HCFCD identifies C-18 Project Hunting and related Hunting Bayou projects, including flood-risk-reduction work involving detention and channel/conveyance improvements.
- Frame this neutrally: the dashboard is designed to help residents understand observed waterway response around a corridor where flood-control investments exist. Do not say the dashboard proves those investments worked.

CRITICAL PRODUCT POSITIONING
Existing public dashboards already show rainfall, water level, channel status, inundation, floodplain context, or community survey results. The value added here is not “new public raw data.” It is an understandable, waterway-specific interpretation of observed behavior alongside a plainly explained infrastructure context.

Use this user-facing framing:
“See how Hunting Bayou is responding, what detention and channel improvements are intended to do, and how Dutch adaptive water-management principles can help residents interpret local flood infrastructure.”

Never claim:
- that a detention basin caused a change shown in the live data;
- that the dashboard predicts flooding;
- that a displayed trend means a person is safe;
- that this replaces HCFCD alerts, emergency management, or hydraulic modeling;
- that data refresh is real-time or guaranteed.

Use the phrase “near-live observations” and show the data retrieval timestamp prominently.

DEPLOYMENT CONSTRAINTS
GitHub Pages is static hosting. It cannot run a persistent backend server. Therefore implement:
- a static public front end in /docs;
- a Python data ingestion and processing pipeline in /scripts;
- a GitHub Actions workflow that runs manually and on a periodic schedule, fetches API data, creates processed JSON/CSV and a low-power HTML snapshot, then deploys the /docs folder to GitHub Pages;
- a last-updated timestamp and stale-data warning in the dashboard.

Run the scheduled workflow hourly at an off-hour minute, for example minute 17, not exactly at the top of the hour. The site must state that scheduled refresh timing is best-effort and not guaranteed.
